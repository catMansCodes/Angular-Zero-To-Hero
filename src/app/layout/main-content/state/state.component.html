1. Component state vs application state

State is the data that drives your UI. In Angular you usually separate local state (one component) from shared state (many routes or features).

Picking the right scope keeps components simple and avoids accidental coupling.

How they differ
  • Component state — short-lived, private to the template and class.
  • Application state — survives navigation and is shared via services, stores, or URL.
  • Clear boundaries make testing and refactoring easier.

2. NgRx basics (store, actions, reducers, effects)

NgRx follows a Redux-style model: a single store, events as actions, pure reducers for transitions, and effects for async work such as HTTP.

The flow is predictable: action → reducer/effect → new state → OnPush-friendly views.

Core pieces
  • Store — single source of truth (observable state).
  • Actions — plain objects describing what happened.
  • Reducers — pure functions: (state, action) → new state.
  • Effects — listen for actions, run side effects, dispatch follow-up actions.

3. Signal-based stores

@ngrx/signals and similar patterns build on Angular signals for fine-grained updates with less boilerplate than classic store setup in smaller and mid-sized apps.

They align with computed() and signal() already used in modern components.

Why teams adopt them
  • Reactive updates without pushing every change through RxJS chains.
  • Often less ceremony than full NgRx for medium complexity.
  • Fits the platform direction (signals-first Angular).

4. Choosing a pattern

There is no single winner: small apps often use services + signals, larger apps may add NgRx or signal stores for stricter rules and devtools.

Match complexity to the team—avoid global stores for data that only one screen needs.

Common options
  • Local state in the component (or inputs/outputs).
  • Shared injectable services holding signals or BehaviorSubjects.
  • Centralized NgRx or NgRx Signal Store for cross-cutting app state.
