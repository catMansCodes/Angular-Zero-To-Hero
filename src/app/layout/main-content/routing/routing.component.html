1. Router configuration

The Angular Router maps URL paths to components. You declare a Routes array (often in app.routes.ts), provide provideRouter / RouterModule, and render matched views in a router-outlet.

Core ideas
  • Routes — path → component (or loadChildren / lazy import).
  • Outlet — where the active route’s component appears.
  • Order — specific paths before wildcards (**).

2. Path, matrix & query parameters

Path params (e.g. profile/:id/:name) are part of the route pattern. Query params follow ? for optional data (filters, serialized payloads). Matrix params use ; on a segment (e.g. /profile;name=Jarvis).

Read them with ActivatedRoute (paramMap, queryParamMap).

3. Lazy loading

Load feature code on demand with loadComponent or loadChildren so the initial bundle stays smaller and startup faster.

  • Smaller first paint; pay for routes when users visit them.
  • Works well with named chunk output in production builds.

4. Guards & resolvers

Guards (canActivate, canMatch, etc.) decide whether a navigation may proceed. Resolvers prefetch data before the route activates (or use fetch in newer APIs where applicable).

  • Auth, roles, feature flags.
  • Preload records so the component opens with data.

5. Nested routing

Child routes can share a parent layout: the parent template holds a primary router-outlet; child routes render inside it. Optional named outlets support multiple parallel targets (e.g. modals sidebars).

6. Wildcards & 404

A ** route catches unknown URLs—typically a “page not found” component. Register it last so it does not swallow real routes.

Try a broken link (opens your 404 page)

7. Programmatic navigation

Inject Router and call navigate() / navigateByUrl() after logins, form saves, or conditional flows. Combine with queryParams, queryParamsHandling, and replaceUrl as needed.
