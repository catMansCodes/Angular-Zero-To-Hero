1. Why use services?

Services hold reusable business logic and data access so components can stay focused on rendering and user interaction.

Typical responsibilities
  • Fetch and update API data.
  • Share state between multiple components.
  • Encapsulate validation and business rules.

2. Dependency injection fundamentals

Angular creates dependencies from its injector instead of classes constructing them directly. This enables loose coupling and easier testing.

  • Inject by constructor or inject() function.
  • Mock services easily in unit tests.
  • Swap implementations with providers.

3. HttpClient & interceptors

HttpClient handles REST calls (GET/POST/PUT/DELETE). Interceptors run around every request/response for cross-cutting behavior.

Common interceptor jobs
  • Attach auth tokens.
  • Global error mapping.
  • Logging and timing metrics.

4. Provider scopes & hierarchy

Service lifetime depends on where the provider is registered: app root, feature route, or component.

  • Root: singleton for the whole app.
  • Feature: one instance per lazy feature boundary.
  • Component: isolated instance per component subtree.

5. Lightweight state with services

For small and medium apps, a service can be enough for shared state before adopting a full store solution (NgRx or signal store).
