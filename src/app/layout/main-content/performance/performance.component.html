1. Change detection strategies

Angular runs change detection to keep the view aligned with application state. With the default strategy, the framework walks the component tree to see what changed.

Knowing when and how often checks run helps you avoid unnecessary work in large UIs.

Key points
  • The default strategy can visit every component in the tree.
  • Heavy templates or frequent updates can make checks costly.
  • Optimization focuses on fewer checks and smaller templates.

2. OnPush change detection

With ChangeDetectionStrategy.OnPush, Angular checks a component mainly when inputs change (by reference), an event originates in the template, or you mark the branch explicitly (for example with markForCheck() or async pipe / signals).

Pair OnPush with immutable patterns or signals so Angular can skip work safely.

Good fits for OnPush
  • Presentational components with clear @Input() boundaries
  • Immutable or observable-driven data
  • Signal-based state where updates are explicit

3. Lazy loading & code splitting

Route-level lazy loading loads feature modules or route components on demand, so the initial bundle stays smaller and Time-to-Interactive improves.

The CLI and esbuild/webpack split chunks per lazy import automatically in production builds.

Benefits
  • Smaller first payload and faster cold start
  • Features pay for their JS only when navigated to
  • Easier scaling as the app grows

4. Tree shaking & bundle size

Tree shaking removes unused exports from the production bundle. Angular’s production build (with optimizations on) applies this together with minification and scope hoisting.

Prefer side-effect-free ES modules and explicit imports so the bundler can drop dead code reliably.

Practices
  • Avoid barrel files that re-export entire libraries if you only need one symbol
  • Use standalone APIs and tree-shakeable RxJS imports (import { map } from 'rxjs/operators')
  • Run ng build --configuration production and inspect chunk sizes

5. Virtual scrolling (CDK)

For very long lists, rendering every row wastes memory and slows change detection. Virtual scrolling (Angular CDK ScrollingModule) recycles DOM nodes and only mounts what is visible plus a small buffer.

Use it for tables, feeds, and pickers where thousands of rows are possible.

Typical use cases
  • Large selectable lists and log viewers
  • Infinite scroll combined with pagination APIs
  • Dashboards with dense tabular data
