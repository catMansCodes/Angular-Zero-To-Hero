1. Unit testing with Jasmine & Karma

Unit tests exercise small pieces of code in isolation—functions, pipes, services, or component classes—so regressions show up before merge.

Jasmine describes specs and expectations; Karma runs them in real browsers (or headless Chrome) via the Angular CLI.

Why it matters
  • Catches bugs close to where they’re introduced.
  • Gives confidence when refactoring.
  • Documents expected behavior for the next developer.

2. Component testing

Use TestBed to create a component fixture, trigger change detection, and query the DOM. You verify bindings, outputs, and how the template reacts to state.

Prefer testing through the public API (inputs, buttons, emitted events) rather than private fields.

Typical checks
  • Class and template logic with fixture.detectChanges().
  • @Input() / @Output() behavior.
  • User flows: click, type, fakeAsync / tick() when timing matters.

3. Service testing & HTTP mocking

Inject services with TestBed.inject() or the component’s injector. For HttpClient, use HttpClientTestingModule and HttpTestingController to assert URLs, methods, and bodies without real HTTP.

Flush pending requests in each test so the next spec doesn’t see leaked calls.

Common scenarios
  • Mock success and error responses.
  • Assert headers, query params, and payloads.
  • Test retry or timeout logic deterministically.

4. End-to-end (E2E) testing

E2E tests drive the app like a user: load pages, click, fill forms, and assert visible outcomes. They validate routing, lazy loading, and integration between layers.

Teams often use Cypress or Playwright; keep E2E suites smaller and focused on critical journeys.

Good candidates
  • Login and checkout flows.
  • Main navigation and deep links.
  • Features that many unit tests would duplicate awkwardly.

5. Coverage & practices

Coverage reports show which lines branches were executed. Use them as a guide—not a goal by itself. Meaningful assertions on business rules beat empty “line touched” tests.

Run ng test --code-coverage (or your CI job) and review HTML reports for untested branches.

Habits that help
  • Cover critical paths and edge cases first.
  • Keep tests independent and fast (no shared mutable globals).
  • Avoid brittle selectors; prefer stable roles, labels, or test ids.
