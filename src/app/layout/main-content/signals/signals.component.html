1. Signals & fine-grained updates

Signals hold reactive state. Reading a signal in a template or in a computed registers a dependency; when you set or update the signal, consumers re-run efficiently—without manual subscriptions.

signal(), computed(), and effect() (for side effects) are the core building blocks. linkedSignal() can derive state from another signal when you need linked lifecycles—see the official docs for API details.

Why teams adopt signals
  • Predictable updates and less RxJS boilerplate for UI state.
  • Works well with OnPush and modern control flow.
  • Composable with computed() for derived values.

2. Signal-based component state

Prefer signals for local UI state: form toggles, selections, and lists. Call .set() / .update() from event handlers; the view stays in sync.

  • Less indirection than manual change detection triggers.
  • Clear data flow: state lives on the class, mutations are explicit.

3. Computed & effects

Computed signals derive values from other signals and cache until a dependency changes. Use effects sparingly for logging, persistence, or non-declarative side work—keep them thin.

Mini demo: counter & doubled (computed)

count = {{ demoCount() }} doubled = {{ demoDoubled() }}
Practices
  • Use computed for anything you can express as a pure function of other signals.
  • Avoid heavy work or HTTP inside effect() without guards.

4. RxJS alongside signals

RxJS remains ideal for asynchronous streams: HTTP, WebSockets, user input debounced into streams, and complex operator chains. You can bridge to signals with toSignal() / toObservable() where needed.

Common operators: map, filter, switchMap, catchError.

5. Observables vs signals

Signals
  • Synchronous reads, great for UI state
  • No subscription lifecycle in templates
  • computed for derived state
Observables
  • Multiple async events over time
  • HTTP, routing events, complex streams
  • Compose with operators
